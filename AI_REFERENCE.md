# Caelix AI Reference

Use this file as the first stop when an AI agent needs to work on Caelix. It is intentionally short; the detailed user-facing docs live under `docs/src/`.

## Project Shape

Caelix is a Rust workspace for a NestJS-style backend framework:

- `crates/caelix-core`: framework primitives: DI container, modules, providers, controllers, responses, exceptions, guards, interceptors, events, request context, logging, and service-level cache.
- `crates/caelix-macros`: proc macros: `#[injectable]`, `#[guard]`, and `#[controller("...")]`.
- `crates/caelix-actix`: Actix Web runtime integration and `Application`.
- `crates/caelix-axum`: Axum runtime integration and Tower-layer escape hatch.
- `crates/caelix`: public framework facade that re-exports core and macros, plus exactly one runtime behind the `actix` (default) or `axum` feature.
- `crates/caelix-cli`: project/code generation CLI.
- `docs/src`: mdBook documentation. Update docs when changing public behavior.

## Core Architecture

Registration is explicit and metadata-driven. Do not add filesystem scanning or hidden auto-registration.

- A module implements `Module` and returns `ModuleMetadata`.
- `ModuleMetadata` chains `.import::<M>()`, `.provider::<T>()`, `.provider_async_factory::<T, _, _>(...)`, `.controller::<C>()`, and event handler registration.
- `try_build_container::<M>()` creates a `Container`, registers imports first, then providers, then controllers, then event handlers, validates declarations, and runs bootstrap hooks.
- Shutdown runs in reverse order: controllers, providers, then imports.
- `Container` stores services by `TypeId` as `Arc<dyn Any + Send + Sync>`. `resolve::<T>()` returns `Arc<T>` and panics if missing; fallible startup paths should use the `try_*` APIs and return `HttpException`.

## Provider Pattern

Prefer `#[injectable]` for normal services:

```rust
#[injectable]
pub struct UserService {
    repo: Arc<UserRepository>,
    logger: Arc<Logger>,
}
```

Rules:

- Injectable struct fields must be `Arc<T>`.
- `Arc<Logger>` is special-cased to `container.resolve_logger("StructName")`.
- Unit structs are supported; tuple structs are rejected.
- Manual `Injectable` impls are fine when construction needs custom async logic.
- Lifecycle hooks live on `Injectable`: `on_module_init`, `on_bootstrap`, `on_shutdown`.

## Controller Pattern

Controllers are generated by `#[controller("/base")]` on an `impl` block. The same source supports the selected Actix or Axum backend.

```rust
pub struct UserController {
    service: Arc<UserService>,
}

#[controller("/users")]
impl UserController {
    #[get("/{id}")]
    async fn find_one(&self, #[param] id: String) -> Result<Response<UserDto>> {
        Ok(Response::Body(self.service.find_one(id).await?))
    }
}
```

Supported route attributes: `#[get]`, `#[post]`, `#[patch]`, `#[put]`, `#[delete]`.

Supported extractor attributes:

- `#[param]`: Actix `web::Path<T>`, passed into the handler as `T`.
- `#[body]`: Actix `web::Json<T>`, passed as `T`.
- `#[query]`: Actix `web::Query<T>`, passed as `T`.
- `#[user]`: reads a cloned value from `RequestContext`; missing values become `UnauthorizedException`.
- `#[validate]`: validates extracted values through `validator::Validate`.

Guards and interceptors can be applied with `#[use_guard(Type)]` and `#[use_interceptor(Type)]` at controller or method level. Controller-level entries run before method-level entries.

## Responses And Errors

Handlers should return values implementing `IntoCaelixResponse`, usually:

- `Response::Body(value)` for `200` JSON.
- `Response::WithStatus(status, value)` for JSON with a custom status.
- `Response::json`, `Response::text`, or `Response::bytes` for raw payloads.
- `Response::no_content()` for `204`.
- `Response::stream` / `Response::sse` / `Response::file` for streaming (`Result<HttpResponse>`).
- `HttpException` types for errors.

`HttpResponse.body` is a `ResponseBody` (`Buffered` or `Streaming`) — breaking vs `Vec<u8>`; use `body_bytes()` / `as_buffered_mut()`. `HttpResponse.headers` is `Vec<(String, String)>` via `with_header` (dynamic values ok; not a full HeaderMap). Actix maps streaming via `.streaming(...)` and applies headers. `EventBus::subscribe::<E>()` creates a live stream; `emit` runs handlers first, then fans out only on success. `StreamExt` is re-exported from `caelix` for stream adapters. Interceptors can rewrite buffered bodies only.

Server errors intentionally hide their internal message and serialize as a generic `Internal Server Error`. Client errors may include validation `errors`.

## Cache Rule

Cache support is explicit service-level caching. `CacheModule` registers `MemoryCacheStore` and `Cache`; services opt in by injecting `Arc<Cache>`.

Do not add automatic HTTP response caching. Users can implement response caching themselves with an interceptor that reads from and writes to `Cache`.

## Public Facade

Most users should depend on `caelix`:

- `caelix` re-exports `caelix_core::*` and `caelix_macros::*`.
- With feature `actix`, it exports `Application`, `TestApplication`, `main`, and `to_actix_response`; with feature `axum`, it exports `Application`, `main`, `to_axum_response`, and `AxumRouterBuilder`.
- `#[caelix::main]` / `#[caelix::test]` and generated controllers expand through hidden runtime re-exports, so apps need only a `caelix` dependency for runtime macros and controller code.
- `caelix::prelude::*` exports core and macros (not `test`/`main`, so Rust’s `#[test]` is not shadowed).

Inside generated macro code, paths intentionally use `caelix::...`; keep that facade stable.

## Tests And Checks

App authors integration-test modules with `TestApplication` (Actix feature):

```rust
#[caelix::test]
async fn should_create_user() {
    let app = TestApplication::new::<AppModule>()
        .override_provider(UserRepository::in_memory())
        .await;

    app.post("/users").json(dto).send().await.assert_status(StatusCode::CREATED);
}
```

Overrides are type-id based (`ProviderOverrides` / `try_build_container_with_overrides` in core). The replacement must be the same concrete type `T` that production injects. Nested imports are covered because overrides live on the container during registration.

Useful commands:

```sh
cargo test
cargo test -p caelix-core
cargo test -p caelix-macros
cargo test -p caelix-actix
cargo test -p caelix-cli
```

Compile-fail expectations use `trybuild` under `crates/*/tests/ui`. If macro diagnostics change intentionally, update the matching `.stderr` files with the normal trybuild workflow.

## Repository Conventions

- Follow the existing explicit registration model.
- Prefer small, direct changes in the owning crate.
- Add or update tests when touching public behavior, macro expansion, lifecycle ordering, cache behavior, or error shapes.
- Update mdBook docs for public API or behavior changes.
- Keep generated examples and docs aligned with the public `caelix` facade instead of internal crate paths unless the docs are explicitly about internals.
